一句话结论

NanoBot 已经有手腕了（run_nong 桥），但还没把 CLI 这只手、Toolkit 这套动作说明书接成完整胳膊。

四个缺口按优先级排：allowlist 缺 lit/slice/progress 是最小最稳马上见效的；SkillLoader 还是全量塞入而不是精准触发+渐进披露；Plugin 安装体系还没做；WebUI 还不是"Toolkit/Nong 控制台"。

1. 三项目分工（说人话版）

Nong.Cli.Net           — 手和工具箱（确定性执行层）
  96 个命令，15 个命令模块，一个 nong.dll
  负责 Word/PDF/OCR/slice/skill/progress 这些命令的确定性执行
  不教 AI 怎么用，只负责把活干完

Nong.Toolkit.Net       — 说明书和动作策略（技能教学层）
  15 个 SKILL.md，告诉 AI 什么时候用哪个命令、怎么读证据、怎么避免乱来
  不放 C# 执行逻辑，只放文档和少量辅助脚本
  核心机制：精准触发 + 渐进披露（先 SKILL.md 入口，再按需加载 reference）

Nong.NanoBot.Net       — 运行时和驾驶舱（Agent 运行层）
  负责模型、会话、工具调用、WebUI、记忆、事件、插件安装、状态展示
  内置 run_nong 桥：用参数数组调用 nong，有 workspace 限制、allowlist、超时、输出截断
  内置 SkillLoader：加载 Toolkit skill 进 Agent 上下文

2. 现有基础：run_nong 桥（已做对的事）

run_nong 桥的方向是对的，底层实现也是扎实的。

它在 Nanobot.Core/Tools/Builtin/NongTool.cs 里，已经做对了这几件事：

  • 参数数组而非 shell 字符串：AI 传 ["pdf", "check", "paper.pdf"]，不由字符串拼接，避免注入和转义问题。
  • workspace 限制：workingDirectory 不能逃出 ~/.nanobot/workspace，防止 AI 乱读系统文件。
  • allowlist 白名单：只有白名单里的 root 命令能执行，不是所有 nong 命令都开放。
  • 超时 + 输出截断：默认 120s 超时、20000 字符截断，防止失控。
  • 结构化错误：missing_args / nong_root_not_allowed / working_directory_not_found / nong_execution_error 都有明确错误码。
  • --json 自动追加：默认 appendJson=true，确保输出是机器可读的 JSON。
  • 空 allowlist 拒绝一切：不会因为配置错误变成全开放。
  • 可配置：通过 config.json 的 tools.nong 段控制 command、timeoutMs、maxOutputChars、allowedRoots。

3. 四个缺口现状

缺口 1：allowlist 缺 lit、slice、progress

NongTool.DefaultAllowedRoots 还是 Nong 3.x 时代的命令面，没跟上 4.0.0。

当前默认白名单有 12 个 root：commands, word, inspect, chart, excel, diagram, genre, icons, skill, pptx, ocr, pdf

Nong 4.0.0 实际有 15 个命令模块。缺了 3 个：lit、slice、progress。

这意味着 NanoBot 明明接了 Nong 4.0.0，却不能调用文献检索（lit）、统一切片（slice）、进度报告（progress）三个完整命令面。

缺口 2：SkillLoader 还是全量塞入，不是精准触发+渐进披露

当前 SkillLoader 只是把 ~/.nanobot/workspace/skills/*/SKILL.md 全部读出来拼成一个字符串塞进上下文。

这和 Toolkit 设计的"精准触发 + 渐进披露"是两回事：

  • 精准触发：AI 应该先看 skill 列表（名字+一句话描述），根据当前任务选择激活哪个 skill。不是一股脑把所有 skill 的完整内容都灌进去。
  • 渐进披露：激活一个 skill 后，先看 SKILL.md 入口（简短），需要时再按 reference 下钻。不是把 references/ 里的文件也一次性全读。
  • 当前实现：SkillLoader.BuildSkillSection() 只读了 SKILL.md 文件内容，没读 references/ 子目录。所以 references 下钻功能根本不存在。
  • Toolkit 侧已经准备好的东西：word 有 7 个 reference、excel 有 3 个、multimodal 有 3 个。这些 reference 文件现在对 NanoBot 是完全不可见的。

缺口 3：Plugin/bootstrap 还在计划里，没实现

以下全部缺失：

  • 没有 nanobot plugin marketplace add/install/status/update 命令。
  • 没有 plugin.json 安装器（指定 skill 来源、版本、依赖）。
  • 没有 nong commands --json capability catalog 集成。NanoBot 不知道已安装的 nong 有哪些命令可用。
  • WebUI 看不到 Toolkit 和 Nong 的安装状态。

缺口 4：WebUI 是可用工作台，但不是"Toolkit/Nong 控制台"

当前 WebUI 能做：聊天、流式、看文件、看工具事件。

当前 WebUI 不能做：

  • 看不到 Nong CLI 是否安装、版本号是多少。
  • 看不到 Toolkit 是否安装、有哪些 skill 已加载。
  • 看不到命令能力是否完整（哪些 root 可用、哪些被 allowlist 挡住了）。
  • 看不到 OCR runtime 是否就绪。
  • 看不到 skill catalog 状态。

4. 施工顺序和理由

第 1 步 · P0 · 最小最稳马上见效

补 NanoBot 的 Nong 4.0.0 命令面：allowedRoots 加 lit、slice、progress

改哪里：

  • Nanobot.Core/Tools/Builtin/NongTool.cs — DefaultAllowedRoots 数组加 "lit"、"slice"、"progress"。
  • Nanobot.CLI/Program.cs — onboard 默认 config 的 allowedRoots 加同样的三项。
  • Nanobot.Tests/ToolTests.cs — 加一个测试：验证默认 allowlist 包含 14 个 root（不含 __render-worker），且 lit/slice/progress 都能通过 IsAllowedRoot。
  • README.md + README.zh-CN.md — 在 Tools 行的 Nong CLI bridge 描述里注明支持的 14 个命令模块。

不改什么：不改 Nong 4.0.0 本身，不改 Toolkit，不改 WebUI。

风险：几乎为零。lit/slice/progress 三个命令模块在 Nong 4.0.0 中已经稳定实现。allowlist 只是放行，不影响已有命令的执行逻辑。

验证：dotnet test 全部通过，新增 allowlist 覆盖测试通过。

第 2 步 · P1 · 把 CLI 能力面变成 NanoBot 可查询的数据

做 nong commands --json capability discovery

改哪里：

  • Nanobot.Core/Tools/Builtin/NongTool.cs — 新增 CapabilityDiscovery 方法，调用 nong commands --json 并解析返回的 96 个命令列表。
  • Nanobot.Core/Models/ — 新增 NongCapability 模型（Root、Command、Description、Parameters）。
  • Nanobot.CLI/Program.cs — 启动时跑一次 capability discovery，把结果缓存进内存，供 WebUI API 查询。
  • Nanobot.Web/ — 新增一个 /api/nong/capabilities 端点，返回 capability 列表。

设计要点：

  • capability discovery 是只读操作，不走 allowlist 拦截（commands 本身已在 allowlist 中）。
  • 缓存策略：启动时跑一次，后续手动刷新。不在每次工具调用时跑。
  • 如果 nong 未安装或版本不对，返回结构化错误而不是崩溃。

不改什么：不改 Nong 4.0.0 的 nong commands --json 输出格式（已经稳定）。

风险：低。只在启动时多跑一个进程，不影响运行时性能。

第 3 步 · P2 · 不再靠手动丢进 workspace/skills

做 Toolkit skill 安装/加载机制

改哪里：

  • Nanobot.Core/Skills/SkillLoader.cs — 重写为两阶段：Phase 1 返回 skill 索引（名字+描述），Phase 2 按需加载指定 skill 的 SKILL.md + references。
  • Nanobot.Core/Skills/SkillPackage.cs — 新增模型：plugin.json 解析、skill 元数据、依赖声明。
  • Nanobot.Core/Skills/SkillInstaller.cs — 新增：从本地目录或 Git 仓库安装 skill 到 workspace/skills。
  • Nanobot.CLI/Program.cs — 新增 nanobot plugin 子命令组：list、install、remove、update、status。
  • Nanobot.Tests/SkillLoaderTests.cs — 扩展：测试两阶段加载、reference 下钻、截断策略。

设计要点：

  • Skill 索引阶段返回简洁列表（名字+一句话描述），不超过 500 字符，不挤占上下文。
  • 激活阶段：AI 通过工具调用 load_skill(name) 来加载具体 skill 的 SKILL.md。
  • 下钻阶段：AI 通过 load_skill_reference(name, ref) 来按需加载 reference 文件。
  • Toolkit 的 plugin.json 格式：{"id":"nong-toolkit","version":"2.4.0","skills":[...],"requires":{"nong":"4.0.0+"}}

不改什么：不改 Toolkit 的 SKILL.md 文件结构和 reference 目录布局（已经规范）。

风险：中。SkillLoader 重写会影响所有依赖 skill 的 Agent 运行。需要充分测试两阶段加载和向后兼容。

第 4 步 · P3 · 一屏看清整条胳膊

做 WebUI 状态面板：Nong.Cli、Nong.Toolkit、OCR runtime、skill catalog、命令能力

改哪里：

  • Nanobot.Web/wwwroot/index.html — 在侧边栏加第 6 个 rail button："状态"（data-view="status"）。
  • Nanobot.Web/wwwroot/app.js — 新增 StatusView 模块：调用 /api/nong/capabilities、/api/nong/version、/api/toolkit/status、/api/skills/catalog 并渲染面板。
  • Nanobot.Web/wwwroot/styles.css — 新增状态面板样式。
  • Nanobot.Web/Program.cs — 注册新的 API 端点。
  • Nanobot.Web/WebContracts.cs — 新增状态相关的 DTO。

面板应展示的内容：

  • Nong.Cli.Net：是否安装、版本号、命令模块数、已启用 root 数。
  • Nong.Toolkit.Net：是否安装、版本号、skill 数量、已加载 skill 数。
  • OCR runtime：是否安装、runtime 包版本、模型状态。
  • Skill catalog：每个 skill 的名字、状态（已加载/未加载/有更新）、CLI 命令映射。
  • 命令能力矩阵：每个命令模块是否在 allowlist 中、是否通过 capability discovery 验证。

不改什么：不改已有的 chat/files/tools/memory/settings 面板功能和样式。

风险：低。新增面板不影响已有功能的 HTML/CSS/JS。API 端点是纯查询，没有副作用。

7. 和现有 Toolkit / CLI 审计结果的对照

这个方案不是凭空想的，是对齐了 Nong.Cli.Net 和 Nong.Toolkit.Net 已有审计结论的。

对照点：

  • Nong.Cli.Net toolkit-vs-cli-full-map.html 已确认：96 个命令、15 个命令模块、100% 技能覆盖率。NanoBot 的 allowlist 应该反映这 15 个模块（除 __render-worker 内部命令外）。
  • Nong.Toolkit.Net toolkit-skill-cli-package-map.html 已确认：skill↔CLI 对应关系、OCR 链的四层映射。NanoBot 的 WebUI 状态面板应该能展示这张映射表。
  • Nong.Cli.Net log/reports/pages/plans-2026-06-10-official-command-surface-alignment.html 已确认：公文命令面已对齐。NanoBot 的 allowlist 不需要额外动作（word/inspect/genre 已在白名单中）。

8. 不在本期范围内的东西（显式排除）

  • 不改 Nong 4.0.0 本身：命令面、输出格式、包结构都已经稳定，不需要动。
  • 不改 Toolkit 的 SKILL.md 文件：skill 内容、reference 结构、plugin.json 格式已规范。
  • 不做 skill-manager 的 CLI 镜像：skill-manager 是治理层 skill，不做成 CLI 命令。保持 Toolkit 现有边界。
  • 不做跨仓库的自动化发布管线：三个仓库各自独立发版。NanoBot 的 plugin install 只做本地安装，不做 NuGet 自动拉取。
  • 不做 OCR runtime 的自动下载：OCR runtime 安装仍由 nong ocr install-model 负责，NanoBot 只展示状态。

附录 A：当前 run_nong 桥完整调用链

AI 模型
  │  调用 tool: run_nong
  │  args: ["word", "read", "paper.docx"]
  ▼
NongTool.ExecuteAsync()
  │  1. ReadArguments() → ["word", "read", "paper.docx"]
  │  2. ResolveRootArgument() → "word"
  │  3. IsAllowedRoot("word") → true（在白名单中）
  │  4. AppendJson → ["word", "read", "paper.docx", "--json"]
  │  5. ResolveWorkingDirectory() → ~/.nanobot/workspace（不能逃逸）
  │  6. ProcessNongCommandRunner.RunAsync()
  │     ├─ Process.Start("nong", "word", "read", "paper.docx", "--json")
  │     ├─ 等待（最多 timeoutMs）
  │     ├─ 超时 → Kill(entireProcessTree: true)
  │     └─ 返回 stdout + stderr
  │  7. Truncate() → 截断过长输出
  │  8. 返回结构化 JSON
  ▼
AI 模型收到 {"tool":"run_nong","exitCode":0,"stdout":"{...}"}
