## Phase 5: 上下文窗口管理

改动: ContextRenderer.GetStableToolSchema()

工具超过 25 个时不再把完整 tool schema 写入系统提示词，改为简短计数行:
```
[125 tools available via function calling]
```

理由: 工具 schema 已通过 function-calling API 的 `tools` 参数发送，系统提示词里的重复列出来浪费 token。
工具 ≤25 个时保持原有行为（完整 schema 内联）。

效果:
- 125 工具注册后系统提示词节省 ~8000 tokens (工具 schema JSON)
- 同时降低 DeepSeek 等模型的首次 token 消耗
- LLM 仍能通过 function-calling 拿到全部 125 工具的 schema

## Phase 6: 确认机制

新增: NongConfirmationHook (IAgentHook)

### 危险命令分类

| 风险等级 | 命令前缀 | 策略 |
|---------|---------|------|
| install | nong_ocr_install_model | 每次确认 |
| camera | nong_ocr_camera | 每次确认 |
| token | nong_ocr_cloud, nong_ocr_to_word | 每次确认 |
| write | nong_word_create/fill/crop/..., nong_pdf_merge/split/..., nong_pptx_create, nong_excel_create/style/..., nong_chart_*/nong_diagram_* | 每 session 每组确认一次 |

### 工作原理

1. AgentLoop 启动时注册 NongConfirmationHook
2. BeforeToolAsync 被调用: 检查工具名是否以 `nong_` 开头
3. install/camera/token → 直接拒绝，返回 "[Nong guard] ... Reply confirm to proceed"
4. write → 检查 session 中该 group 是否已确认 → 未确认则拒绝
5. LLM 收到拒绝消息 → 向用户解释并要求确认
6. 用户说 "confirm" → LLM 重新调用工具 → BeforeToolAsync 放行
7. AfterToolAsync: write 命令执行成功后自动标记该 group 为已确认

### 集成

- CLI Program.cs: Agent 构造时传入 NongConfirmationHook
- Web Program.cs: 同上
